# PRD 生成 Skill（墨刀 D2D 框架版）

> 目标：用于后续在其他项目中“一键生成 PRD 文档”，产出结构与写作口径对齐本仓库 `modao-d2d/prd/端到端 PRD.md` 的框架与风格，并复用本项目多轮迭代沉淀的交互与数据规范。

---

## 0. 使用方式（给生成器/Agent 的指令）

- **输出文件**：默认生成 `prd/端到端 PRD.md` 同结构的 PRD（可改为项目名）。
- **语言**：中文为主；如需海外版/英文版，按“海外版适配”章节输出对照说明。
- **写作约束**：
  - 不写“空话、套话、泛泛而谈”；每段必须回答“做什么/为什么/怎么做/边界是什么/如何验收”。
  - 表格优先（信息密度高、便于研发对齐）。
  - 关键交互必须给“状态、触发、反馈、异常、权限、埋点”。
  - 文档内若涉及英文文案，必须同步给出英文版本，避免后续补翻译。

### 0.1 版本记录规范（必须）

- 在 PRD 顶部维护版本表（如 `V1.0 / V1.1 / V1.2`）。
- 每个版本说明必须写清：
  - 本次更新涉及的模块（例如：首页入口、发布、导出、私有化、英文版）。
  - 关键变更点（最多 3-6 条，按优先级排序）。
  - 与研发强相关的行为变化（如“按钮置灰不可点”“菜单顺序调整”）。
- 若只是文案或体验细节调整，也要单独列条（例如“分享文案优化”）。

---

## 1. 文档结构（必须与 D2D PRD 框架一致）

PRD 必须包含并按顺序输出（标题可微调，但层级一致）：

1. **背景与目标**
2. **用户与场景**
3. **信息架构 & 页面结构**
4. **核心流程（端到端链路）**
5. **关键模块 PRD（逐模块）**
6. **适配说明（私有化 / 海外版等）**
7. **数据与埋点**

> 若项目不需要某章节：保留标题并写“本期不做/不适用 + 原因 + 后续可能性”，不要直接删除章节。

### 1.1 标题层级与目录可见性规范（必须）

- 为保证预览页左侧目录可见，关键功能小节至少使用 `####` 级别。
- 不建议大量使用 `#####` 作为关键功能节点，否则在部分目录组件中不会展示。
- 推荐层级：
  - `###`：页面/区域级模块（如“3-2-4 右侧编辑展示区”）
  - `####`：功能项（如“4）截图优化”“7）发布”）
  - `-` 列表：交互细节与状态规则

## 2. “端到端生成”模块写法（本项目沉淀的关键点）

### 2.1 端到端链路必须闭环

写清楚从入口到交付的闭环：

- **入口**：二级 Tab（如“生成应用”）+ 通用模式命中（系统路由）+ 其它入口（如截图优化）。
- **生成**：流式输出、完成态、失败态。
- **预览/编辑/代码**：同一份产物在不同工作模式下的查看与操作。
- **发布**：发布/更新/撤回的动作与结果。

### 2.2 每个模块必须给“五要素”

对任意模块（示例：发布、截图优化、设备切换、地址栏）必须写：

- **触发条件**：从哪里进入、点击/快捷键/系统自动触发
- **核心状态**：未开始/进行中/成功/失败/禁用/无权限
- **反馈**：loading、toast、hover tooltip、空态、错误提示
- **异常与边界**：网络失败、重复点击、取消、不可用模式、输入校验
- **权限与可见性**：仅自己/公开、编辑权限等（如适用）

### 2.3 交互细节风格（对齐本项目演示 Demo）

生成器写 PRD 时应优先采用以下交互风格描述（用于研发复刻 Demo）：

- **按钮状态驱动**：按钮文案与禁用态随状态变化（如发布/更新/撤回）。
- **即时反馈**：关键操作必须有 toast/状态标记（成功/失败/取消）。
- **互斥下拉**：同一时间只允许一个下拉展开；点击另一个应“立即切换”。
- **预览设备**：PC/Pad/Mobile 预览区域遵守最大宽高限制；非 PC 模式居中展示。
- **单页/多页差异**：单页下载不需要下拉；地址栏下拉单页只展示一个页面。

### 2.5 首页“AI生成应用”入口规范（强约束）

用于后续复用“推荐 + 生成应用 Tab”的一致交互：

- 一级 Tab 保持 `生成应用`。
- 推荐区与生成应用区统一采用命名：
  - `AI生成应用`
  - `AI生成Web应用`
  - `AI生成App`（避免“App应用”重复表述）
- 输入框策略：
  - 选中任一入口都显示胶囊标签（tag pill）。
  - `AI生成应用` 使用通用提示词，不注入特定端型模板。
  - `AI生成Web应用` / `AI生成App` 使用端型提示词。
- 高级配置按钮（设计系统/主题色）显示规则：
  - 选中 `AI生成应用`、`AI生成Web应用`、`AI生成App` 三者任一时均可显示并可配置。

推荐提示词模板：

- 中文：
  - `AI生成应用`：`请描述你的应用需求，我将生成完整的应用和代码...`
  - `AI生成Web应用`：`请描述你的 Web 应用需求，我将生成完整的应用和代码...`
  - `AI生成App`：`请描述你的 App 需求，我将生成完整的应用和代码...`
- 英文：
  - `AI生成应用`：`Describe your app idea, and I will generate the full app and code...`
  - `AI生成Web应用`：`Describe your Web app idea, and I will generate the full app and code...`
  - `AI生成App`：`Describe your App idea, and I will generate the full app and code...`

### 2.4 PRD 文档阅读体验规范（沉淀的高级交互）

生成的 PRD 静态页面或展示框架，应遵循以下阅读体验规范：

- **结构化常显目录 (Sticky Sidebar)**：
  - 放弃浮层遮挡：目录展开后，采用**同级推挤布局 (Push Layout)**，平滑挤压右侧正文区，绝不遮盖阅读内容。
  - 滚动分离：左侧目录栏在屏幕内常驻（Sticky），用户滚动右侧长文档时，左侧目录不随之滚动。
  - 明确的开关：提供清晰的“≡ 目录”与“× 关闭”按钮，点击图标以外区域**不应**意外收起目录，防止打断阅读状态。
- **宽屏适配与阅读舒适度**：
  - 正文区应设置合理的最大宽度（如 `1200px`），并在宽屏下居中显示，避免行长过长导致视觉疲劳。
- **双向阅读联动 (Scrollspy)**：
  - 正文滚动时，左侧目录的对应章节标题需实时**高亮跟随**，让用户明确当前阅读位置。
  - 当高亮项超出目录面板可视区时，目录面板应自动居中滚动至该高亮项。
- **视觉层次与光影克制**：
  - 浅色模式下，弱化或移除大范围的弥散阴影，仅保留极轻微的边界立体感（如 `0 4px 12px rgba(0,0,0,0.03)`），使界面清爽融于背景。
  - 暗色模式下，适当保留阴影与发光效果以拉开 Z 轴空间层级。

## 3. 英文化与多端拆分（可选模块，但写法需一致）

### 3.1 海外版（英文 UI）输出要求

- 必须说明“英文版为独立演示入口/路由”，避免影响中文主流程。
- 文案翻译要求：短、业界一致、可交付研发；不使用生硬直译。
- 说明“同交互、同信息结构，仅替换文案”。
- 对一级 Tab 名称增加“单行显示约束”：
  - 英文文案过长时优先缩短翻译（如 `Marketing` 而不是 `Marketing & Media`）。
  - 不通过随意换行破坏导航一致性。

### 3.2 移动端 Demo（独立模块）输出要求

- 移动端应独立成模块（独立目录/入口），便于后续单独迭代。
- 写清楚移动端范围边界（仅 preview/publish 或其它裁剪）。

## 4. 数据与埋点（必须按“规范格式”输出）

### 4.1 埋点命名规范（必须遵守）

- **事件命名**：`MD_ai_操作对象_操作类型`
- **操作类型**：优先使用规范枚举（`click` / `show` / `enter` / `set` / `add` / `delete`）
- **属性命名**：优先复用常见属性（`scene_tag` / `type` / `name` / `source` 等）
- **属性必须包含中文名**：每个属性必须写“属性英文名（属性中文名，类型）：示例/说明”

### 4.3 看板指标（解释逻辑必须清晰）

- **端到端成功率（价值闭环）**：`prompt_click` → `generation_show(success)` → `publish_show(action=publish,success)` 转化率
- **生成质量**：`generation_show.result` 成功率 + `duration_ms` P50/P90/P99
- **截图优化价值**：
  - 使用量：`screenshot_add`
  - 效果：对比 `prompt_click.source=screenshot` vs `source=d2d_tab/common` 的生成成功/发布成功差异（必要时再拆 `d2d_tab` vs `common`）

> 说明：若已有 `MD_ai_agent_use` 统计二级 Tab 点击，不与 D2D 链路埋点冲突；前者用于“入口选择”，后者用于“链路闭环与质量”。

## 5. 输出模板（给自动生成器的骨架）

生成 PRD 时，建议先输出目录（TOC），再按章节填充。每个模块最少包含：

- **需求描述**
- **用户价值/卖点**
- **交互与状态**
- **异常/边界**
- **验收标准**
- **埋点（若是核心链路）**
- **中英文文案（若涉及英文版）**

---

## 6. 校验清单（生成后自检）

- **结构**：是否严格遵守 1~7 章节与层级
- **目录**：关键小节是否至少为 `####`，可在预览目录中展示
- **链路**：是否存在端到端闭环（入口→生成→预览/编辑/代码→发布）
- **交互**：是否包含状态、反馈、异常、权限
- **文案一致性**：
  - 中文与英文是否同步更新
  - 首页入口命名是否统一（AI生成应用 / AI生成Web应用 / AI生成App）
  - 提示词是否采用“完整的应用和代码”口径
- **导出菜单**：是否明确“导出至墨刀设计”第一项，“导出至墨刀原型”第二项且置灰
- **发布与分享**：是否补充体验性文案优化项到版本日志
- **埋点**：
  - 事件命名是否 `MD_ai_*`
  - 属性是否都带中文名
  - 是否能回答“用不用/成不成/交不交付/截图价值”

### 6.1 PRD 与预览页同步检查（新增）

用于保证“文档改了，演示也同步”：

- 若 PRD 调整了入口名称/顺序，必须同步检查前端配置常量（如 Tab 数组、卡片数组、翻译字典）。
- 若 PRD 调整了“可用/置灰”状态，必须同步更新 UI 组件的 `disabled` 和交互行为。
- 若 PRD 新增英文文案，必须同步更新英文字典映射，不允许回落显示中文。
- 若存在静态 `index.html` 入口页，需同时满足：
  - 本地 `file://` 直接打开可跳转开发服务。
  - 线上（Vercel）使用相对路径可正常访问。
