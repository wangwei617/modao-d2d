## 0. TL;DR（一页带走）

- **分析框架**：**拉新 → 激活 → 留存 → 付费** 四维漏斗；去掉 Referral（待核心漏斗稳定后再纳入）。
- **北极星指标**：**L3 强激活率**（首 7 日内完成「导入墨刀 / 生成分享链接 / ≥3 轮迭代或局部重绘」任一行为的新用户占比）。该指标直接与营收主通道（转项目、席位、订阅雏形）挂钩。
- **激活三级梯度**：L1（发送 prompt）→ L2（首任务完成 + 沙盒交互）→ L3（进入墨刀商业主通道）。
- **付费两阶段**：
  - **现阶段（纯消耗品 / 积分包）**：看 ARPU、档位分布、复购、触发源分布。
  - **未来阶段（订阅 + 消耗品混合）**：看 MRR、Churn、升档、追加积分率。
- **特殊分群**：「新用户拉新漏斗」与「老墨刀用户存量激活漏斗」**分开看**，避免存量自然流量稀释投放 ROI 判断。
- **关键埋点缺口（P0）**：一级 Tab、首条 `user_message` 明文、付费触发源、导入墨刀事件、沙盒交互事件、局部重绘事件、支付订单与用户关联。
- **数据平台分工**：**神策**负责漏斗 / 留存 / 路径事件（主战场），**Metabase** 负责订单与业务表 JOIN（补充），详见 §1.4。
- **反馈体系并行**：定性反馈与满意度单独成册于 [墨刀AI用户反馈与满意度体系_V1.md](./墨刀AI用户反馈与满意度体系_V1.md)，与本文双轮驱动。
- **落地节奏**：S1 骨架盘点（3–5 天）→ S2 埋点补全（1–2 周）→ S3 分群切片 → S4 体验归因 → S5 实验循环 → S6 看板日常化。

## 1. 总体框架

### 1.1 主漏斗

```
[拉新 Acquisition]
  曝光 → 落地页点击 → 注册提交 → 注册成功 → 首次进入工作台
        │
        ▼
[激活 Activation] ★ 核心战场
  L1 发送 prompt  →  L2 首任务完成 + 沙盒交互  →  L3 进入商业主通道
                                                   ├─ 导入墨刀编辑器
                                                   ├─ 生成分享/预览链接
                                                   └─ 多轮迭代 ≥3 轮 或 局部重绘
        │
        ▼
[留存 Retention]
  D1 / D7 / D30 回访 → 首周任务数 → L3 复现次数 → 周活跃天数
        │
        ▼
[付费 Revenue]
  【现阶段】触发源 → 付费墙曝光 → 定价页到达 → 档位选择 → 支付成功
  【未来阶段】订阅转化 + 追加积分 + 升档扩展
```

### 1.2 四条横向切分轴

主漏斗的每一环，都要再按下述维度切片，把"数字"翻译成"人"与"原因"：

| 切分轴 | 维度 | 数据来源 |
|---|---|---|
| **用户分群** | 新用户 / 老墨刀用户、付费 / 免费、行业线（从 `name` 聚类反推） | `user_id`、`org_cid`、订单表 |
| **意图场景** | Top4（大屏/C 端/后台/复刻）/ PRD / 生图 / 其他 / 泛需求 | `mode`、`name`、`plan` |
| **渠道来源** | SEM / 内容 / 主站跨产品 / 分享链接回流 / 直接访问 | UTM 参数、Referrer、入口事件 |
| **体验信号** | 任务耗时、澄清轮数、Token 用量、错配纠偏、报错码 | `total_tokens`、`status`、埋点事件 |

### 1.3 北极星与护栏

| 类型 | 指标 | 说明 |
|---|---|---|
| **北极星** | **新用户 7 日内 L3 强激活率** | 直接预测未来营收，覆盖"拿到价值 + 进入墨刀主通道"双重含义 |
| **营收主指标** | ARPU（现阶段）/ MRR（未来阶段） | 随付费模式演进切换 |
| **护栏 1** | `finished` 完成率 | 不得因增长实验而下降 |
| **护栏 2** | 错配率（R1） | 不得因扩容卡片而上升 |
| **护栏 3** | 首任务 P50 耗时 | 不得因深度引导而显著增加 |

### 1.4 数据平台分工（神策 × Metabase 双工具）

本文不强求所有分析集中在单一工具，采用**神策 + Metabase 双轨**：

| 工具 | 擅长 | 承担环节 | 典型看板 |
|---|---|---|---|
| **神策（主战场）** | 前端事件流、漏斗、留存、用户路径、A/B 实验 | §2 拉新、§3 激活、§4 留存、§5 付费触发与漏斗 | 看板 1/3/4/5 的事件层 |
| **Metabase（补充）** | SQL 查询、跨业务表 JOIN、订单/用户/会话元数据 | 订单与 ARPU、复购分析、任务元数据（`mode`/`token`）、反馈 × 转化交叉 | 看板 2/6 + 反馈专属看板 7~9 |

**神策数据进入 Metabase 的三种方案**（按成本从低到高）：

1. **轻量起步（推荐 MVP 阶段）**：神策计算好的关键节点指标（L3 激活人数、付费转化率等）通过每日导出或 API 同步到数仓汇总表 `fact_daily_metrics`，Metabase 直接查这张表。
2. **中等升级（S3 阶段准备）**：神策原始事件通过"数据订阅/导出"同步到数仓，Metabase 可自行复算漏斗并与订单表 JOIN。
3. **重型规划（6~12 月后）**：神策埋点事件 + 业务表 + 工单表统一进数仓 → 上层指标层（dbt/Superset Metrics）→ 神策/Metabase 共享口径。

> **关键约定**：`user_id` 与 `org_cid` 在神策、Metabase、工单系统中**必须全局一致**，这是跨源打通的最小必要条件。

### 1.5 与《用户反馈与满意度体系》的关联

本文聚焦**定量数据**，定性反馈与满意度评价单独成册于 [墨刀AI用户反馈与满意度体系_V1.md](./墨刀AI用户反馈与满意度体系_V1.md)。

双文档协作关系：

| 本文（定量） | 反馈文档（定性） |
|---|---|
| 回答"**哪里掉了（Where）**" | 回答"**为什么掉（Why）**" |
| 神策漏斗 + Metabase 业务指标 | 产品内嵌评价 + 网易七鱼工单 |
| 看板 1~6 | 看板 7~9（反馈总览 / 反馈 × 漏斗交叉 / 工单深度洞察） |
| 执行节奏 §8 S1~S6 | 采集 MVP 与反馈文档 §9 协同落地 |

**关键协作约定**：
- **埋点 PRD 合并评审**：本文 §6 与反馈文档 §6.4 的神策事件**同一轮评审**，避免字段冲突与重复。
- **看板同期上线**：看板 1~6 与 7~9 尽量同期部署，上线首日即可做"反馈 × 漏斗"交叉。
- **实验护栏叠加**：增长实验（S5）上线时，反馈周报同步观察相关环节满意度评分，作为实验护栏信号。

## 2. 拉新 Acquisition

### 2.1 边界定义

- **起点**：投放创意/搜索结果/外部链接曝光。
- **终点**：**"注册成功 + 首次进入工作台首页"**（不是"注册成功"就止步，避免高漏字引导吃掉真实到达率）。
- **两条并行漏斗**：
  - **新用户漏斗**（目标：增长与投放 ROI）
  - **老墨刀用户首次用 AI 漏斗**（目标：存量激活、跨产品渗透）

### 2.2 指标字典

| 指标 | 定义（分子 / 分母） | 说明 |
|---|---|---|
| **曝光** | 展示次数 | 按渠道/创意拆 |
| **落地页点击率** | 落地页 UV / 曝光 | 落地页含投放页与官网首页 |
| **落地页跳出率** | 无任何交互即离开的 UV / 落地页 UV | 反映文案与真实需求的匹配度 |
| **首屏 CTA 点击率** | "立即体验 / 免费试用"点击 UV / 落地页 UV | CTA 文案可实验 |
| **注册到达率** | 注册页 UV / 落地页点击 UV | 衡量落地页→注册的衔接 |
| **注册完成率** | 注册成功 UV / 注册页 UV | 过程验证码、手机号等摩擦点 |
| **工作台到达率** | 进入工作台 UV / 注册成功 UV | ★ 捕捉"注册完成≠真正进入产品"的漏损 |
| **端到端注册转化率** | 工作台到达 UV / 落地页点击 UV | 拉新总转化 |
| **CAC（获客成本）** | 渠道花费 / 端到端注册成功用户数 | 按渠道拆 |

### 2.3 体验信号

- 落地页加载性能（LCP、FID、CLS），移动端尤其关注。
- 案例/Demo 视频播放完成率。
- "免登录先试一条 prompt" 这类钩子的点击率与后续注册转化（若产品有此设计）。
- 注册验证码发送成功率、验证失败重试次数。

### 2.4 重点诊断问题

1. 投放页宣传的"一句话生成原型"**是否与 Top4 真实场景（大屏/C 端/后台/复刻）对齐**？不对齐会导致高点击低转化。
2. 各渠道的 **端到端转化率 × CAC** 矩阵中，哪些渠道"便宜但水"、哪些"贵但准"？
3. 注册链路中**验证码/手机号/第三方登录** 各自的放弃率分布。
4. **老墨刀用户**在主站哪个入口跳转到 AI 的转化最好（白板 AI、PPT 美化、原型编辑器 AI、首页入口）？

## 3. 激活 Activation ★

> 这是墨刀 AI 最特殊也最关键的一段。**不能以"完成首任务"为终点**——`finished` 已经 93.74%，优化空间小但更关键的是：**完成 ≠ 拿到价值 ≠ 进入商业主通道**。

### 3.1 激活三级梯度

| 梯度 | 触发条件（任一满足） | 产品含义 | 营收关联 |
|---|---|---|---|
| **L1 弱激活** | 发送过至少 1 条 prompt 且任务进入生成流程 | 产品被打开，有意图 | 弱 |
| **L2 中激活** | 首任务 `status=finished` **且** 在沙盒内有交互动作（点击/滚动/查看代码/切换多文件页面） | 看到产出、产生 Aha Moment | 中 |
| **L3 强激活** ★ | 首 7 日内完成任一：<br>① 点击"导入墨刀"并成功保存项目；<br>② 生成可外部访问的分享/预览链接；<br>③ 同一会话内多轮迭代 ≥3 轮 或 触发过至少 1 次局部重绘 | 进入商业主通道、建立依赖 | **强，北极星** |

**为什么是 L3**：L3 行为直接对应 BRD 第 2 条"补全工具链环"与 PRD R3"转项目/席位"的营收动脉。**历史数据会验证：L3 用户的 D30 留存率与付费转化率应显著高于 L1/L2**——若验证成立，所有激活优化都应收敛到提 L3。

### 3.2 指标字典

| 指标 | 定义 | 口径说明 |
|---|---|---|
| **L1 激活率** | L1 用户 / 工作台到达 UV | 首次会话窗口内 |
| **L2 激活率** | L2 用户 / L1 用户 | 首次会话窗口内 |
| **L3 激活率** | L3 用户 / L2 用户 | **首 7 日内**，滑动窗口 |
| **端到端激活率** | L3 用户 / 工作台到达 UV | 北极星指标 |
| **首任务完成率** | 首任务 `finished` / 首任务发起 | 按 `mode`、按场景拆 |
| **沙盒交互深度** | 首任务完成后沙盒内交互事件数 P50 / P90 | L2 的细化 |
| **首次导入墨刀成功率** | 点击"导入"→ 编辑器内保存 / 点击"导入" | 转项目漏斗核心 |
| **局部重绘使用率** | 使用过局部重绘的用户 / L2 用户 | BRD 差异化能力验证 |
| **澄清穿透率** | `clarifying`→`finished` / 进入 `clarifying` | 对应 R5 |
| **错配纠正率** | 用户点击"切换到 X" / 系统提示切换弹出 | 对应 R1 |

### 3.3 体验信号

- **首任务等待时长 P50/P90**：与 express 心智（89.3%）比对，超出预期是流失点。
- **首任务质量代理**：二次修改次数、"重新生成"点击率、复制/下载点击。
- **满意度代理**：是否有 👍/👎、是否主动修改标签、是否主动改 prompt 重发。
- **意图识别失误**：「其他」桶（8.61%）内的首任务占比，这批用户的 `finished` 仅 78%，拉低首激活整体水位。

### 3.4 重点诊断问题

1. **激活最大掉点在哪？** L1→L2 还是 L2→L3？大概率是 L2→L3（看到产出后"没下一步"）。
2. **L3 三条路径哪条最顺？** 导入墨刀 vs 分享链接 vs 迭代/重绘，各自转化率差多少？权重投入往最顺的那条倾斜。
3. **新用户首任务选哪类场景，L3 转化最高？** Top4 各场景是否有显著差异？这决定首页占位示例（R4）往哪类倾斜。
4. **挂载二级标签 vs 不挂载（general 75.19%）**，哪种首任务的 L3 转化更高？若 general 转化反而更高，说明卡片体系反而是"噪声"，要重新想 4.0 的价值点。
5. **错配用户（特别是 `image_generation` 55.3%、`product_planning` 42.6%）**的 L3 转化率是否被显著拖累？如是，R1 错配治理应该和激活实验同步上。

### 3.5 与 PRD 对应

- R1（错配治理）→ 护栏与错配纠正率
- R2（其他桶）→ L1→L2 转化
- R3（Top4 深化）→ L2→L3 转化
- R4（general 路径）→ L1→L2 转化
- R5（澄清体验）→ 澄清穿透率

## 4. 留存 Retention

### 4.1 边界定义

- **起点**：用户完成 L1（首次会话发起）
- **窗口**：D1 / D7 / D30 三档 + 首 30 天累计行为

### 4.2 指标字典

| 指标 | 定义 | 重点 |
|---|---|---|
| **D1 回访率** | 首次激活后次日仍有任务的用户 / 当日激活用户 | 即时粘性 |
| **D7 回访率** | 首次激活后 7 日内回访的用户 / 当日激活用户 | 周内习惯形成 |
| **D30 回访率** | 首次激活后 30 日内回访的用户 / 当日激活用户 | 月度留存 |
| **首周任务数分布** | 首 7 日内任务数 P25/P50/P75/P90 | 找"留存拐点"用（见 4.3） |
| **L3 复现次数** | 首 30 日内触发 L3 的累计次数 P50/P90 | 比"是否触发过 L3"更重要 |
| **周活跃天数** | 周内有任务的天数分布 | 习惯强度 |
| **项目数增长** | 每用户累计导入墨刀的项目数 | 营收侧领先指标 |
| **WAU / MAU** | — | 判断"一次性工具"还是"常用工具" |

### 4.3 关键分析方法：留存拐点分析

**目的**：找到"**首周做到什么程度，D30 留存显著提升**"的行为门槛。

**方法**：
1. 取近 60 天激活的用户作为队列。
2. 按首 7 日内各行为的发生次数分桶（任务数、L3 次数、导入次数、分享次数）。
3. 计算每个桶的 D30 留存率，找拐点。
4. 把拐点翻译为产品目标（例："首周任务数 ≥ 3 次的用户 D30 留存比 < 3 次高 3 倍"→ 产品首周引导目标就是"让新用户完成 3 次任务"）。

### 4.4 流失分析

- **流失用户的最后一个事件是什么？** 错配？澄清超时？报错？Token 耗尽？付费墙拒付？
- **流失曲线形态**：是第 1 天就掉，还是 D7~D14 明显掉？不同形态对应不同问题（一次性 vs 价值稀释）。

## 5. 付费 Revenue

### 5.1 两阶段差异化设计

| 阶段 | 计费形态 | 分析重心 | 核心指标 |
|---|---|---|---|
| **现阶段** | 纯消耗品（积分包一次性付费，4 档：¥68/¥168/¥299 组合/¥? 大包） | ARPU、档位分布、**复购**、触发源 | 付费率、ARPU、复购间隔、¥299 组合档占比 ★ |
| **未来阶段** | 订阅（高中低三档月度会员） + 追加积分购买 | MRR、Churn、升档、追加 | MRR、月 Churn、Low→High 升档率、订阅中追加积分比率 |

> ¥299 档（10000 积分 + 原型 1 年）是订阅雏形，**其占比是未来推包月会员前最重要的 PMF 领先指标**——该占比越高，说明用户对"AI + 墨刀生产力"组合价值认可越强。

### 5.2 现阶段：付费漏斗（双子漏斗）

**子漏斗 A：触发与进入**
```
付费触发条件满足 → 付费墙曝光 → 点击进入定价页
```

**子漏斗 B：定价页到支付成功**
```
进入定价页 → 选档位 → 选支付方式 → 进入结算 → 支付成功
```

### 5.3 关键维度：付费墙触发源

| 触发时机 | 用户心态 | 预期转化 | 分析优先级 |
|---|---|---|---|
| 首次积分耗尽（被动） | "刚用起来就要收费" | 低 | 高（量大） |
| 任务完成后点"导入墨刀" | "已拿到价值想保存" | **高** ★ | **极高**（营收主通道） |
| 任务完成后点"分享/发布" | "想给人看" | 中高 | 高（兼顾裂变） |
| 点"深度思考"或上传附件 | "要做更专业的事" | 中 | 中 |
| 主动访问"升级"入口 | "在比较档位" | **最高** | 中（量少但优质） |
| 复购老用户看到限时特惠 | "要不要囤积分" | 中 | 中 |

**强烈建议**：每个支付订单都落 `trigger_source` 字段，一个字段能回答 80% 的付费优化问题。

### 5.4 指标字典

| 指标 | 定义 | 阶段 |
|---|---|---|
| **付费率** | 付费用户 / 激活用户 | 双阶段 |
| **ARPU** | 支付总额 / 付费用户数 | 现阶段 |
| **档位分布** | 各档订单数占比 | 现阶段 |
| **¥299 组合档占比** | ¥299 订单 / 总订单 | 现阶段（PMF 领先指标）★ |
| **复购率** | 30/60/90 天内二次付费用户 / 首次付费用户 | 现阶段 |
| **复购间隔中位数** | 两次付费间隔的 P50 | 现阶段 |
| **倒计时弹性** | 倒计时状态下支付数 / 非倒计时状态支付数 | 现阶段 |
| **MRR / ARR** | — | 未来阶段 |
| **月 Churn** | 月内取消订阅用户 / 月初活跃订阅用户 | 未来阶段 |
| **升档率** | 低档→中档 / 低档活跃用户 | 未来阶段 |
| **追加积分率** | 订阅期内额外购买积分的订阅用户 / 订阅用户 | 未来阶段 |
| **LTV / CAC** | 用户终身价值 / 获客成本 | 双阶段（未来阶段更准） |

### 5.5 体验信号

- 付费墙出现时用户所处状态：**失败中弹付费墙是致命伤**（例：生成报错后立即弹"升级解锁"）。
- 定价页停留时长、功能对比表滚动深度、FAQ 点击。
- 权益列表中各项的悬停/关注度（验证用户冲哪个权益付费）：当前付费页把"导入至墨刀"放在权益列表最底部，字号一致——这与它本应是营收主通道的定位不匹配，值得 A/B 实验。
- 支付方式失败率（扫码支付 vs 对公支付）。

### 5.6 重点诊断问题

1. 各 **触发源** 的支付转化率差多少？"导入墨刀触发"是否确实转化率最高？
2. **¥299 组合档** 占比多少？用户是真的冲"原型 1 年"来的，还是只因为 83 折对比下 ¥299 看起来划算？（交叉看有没有主动使用过墨刀原型编辑器能回答）
3. **复购率与复购间隔**：短间隔（≤7 天）复购是"算错了额度"还是"真的在高频使用"？
4. **倒计时** 真的在推高转化吗？剔除倒计时的基线转化是多少？
5. **激活梯度 × 付费率** 交叉：L3 用户的付费率是否显著高于 L1/L2？若是，营收最高 ROI 的动作就是"推更多用户到 L3"。

当前 CSV 与线上埋点 **缺失** 的关键字段：

| 字段 | 用途 | 优先级 |

| `l1_tab` | 一级 Tab（推荐/原型设计/产品策划...）点击路径 | **P0** |

| `mode_final` | 发送前最终生效的 `mode`（经错配纠偏后） | P0 |
| `mode_corrected` | 是否触发过错配提示与用户是否采纳 | P0 |

| `sandbox_interaction_*` | 沙盒内的点击/滚动/切换页面事件 | **P0**（L2 判定依赖） |
| `partial_regen_used` | 局部重绘触发事件 | **P0**（L3 判定依赖） |

| `share_link_*` | 分享链接生成 / 对外打开次数 | **P0**（L3 判定 + 裂变预埋） |

| `paywall_exposure` / `paywall_click` | 付费墙曝光与点击 | **P0** |
| `order.trigger_source` | 订单表扩字段：本次订单的触发源 | **P0** |
| `order.user_id` × `org_cid` | 订单与用户、空间关联 | **P0** |
| `signup_source` / `utm_*` | 注册来源、渠道 | P1 |
| `enterprise_first_user_at_ai` | 老墨刀用户首次使用 AI 的时间 | P1 |

### 6.2 埋点工具归属

上述字段**全部在神策完成埋点**，神策是事件流的主战场：

| 事件类型 | 上报载体 | 负责 |
|---|---|---|
| 前端用户行为（点击 / 曝光 / 路径 / 沙盒交互） | 神策 JS SDK | 前端 |
| 后端关键业务事件（导入成功、支付成功、订单变更） | 神策后端 SDK / Webhook | 后端 |
| 订单 / 会员 / 空间主数据 | MySQL / 业务数仓 | 后端 + 数据 |
| 客服工单数据 | 网易七鱼 API 同步至数仓 | 数据 |

### 6.3 数仓建议

1. **订单表 × 用户表 × 任务表 可关联**：脱敏与合规前提下，建立 `user_id` / `org_cid` 的统一主键，三方工具（神策 / Metabase / 网易七鱼）全局一致。
2. **关键事件表**：至少维护 `event_name`, `user_id`, `session_id`, `created_at`, `properties(JSON)`，把上述 P0 事件统一落神策，同时按需同步到数仓。
3. **会员/订阅表**（未来阶段准备）：字段建议 `sub_id`, `user_id`, `plan`, `start_at`, `end_at`, `renewed_from_sub_id`, `churn_reason`。
4. **每日汇总表**（推荐 MVP 阶段就建）：`fact_daily_metrics`，存储神策计算好的关键节点指标（L3 人数、付费转化率等），供 Metabase 直接查询，避免两边算法口径漂移。

## 8. 执行节奏

### S1：漏斗骨架盘点（3–5 天）

- 用**神策已有事件 + Metabase 任务/订单表**，粗跑一版四维漏斗：拉新段大概率缺投放归因数据，激活段用 `status`/`mode`/`token` 近似，留存段用 `user_id` 做简单回访，付费段用订单表。
- **产出**：一份"哪些数字能算、哪些算不出来"的 **缺口清单**，直接对应 §6.1。

### S2：埋点补全（与 S1 并行，1–2 周）

- 优先补 §6.1 中 P0 字段，**全部在神策实现**。
- 同步补 §6.3 中的 `fact_daily_metrics` 每日汇总表，打通神策 → Metabase。
- 与 [反馈体系 §6.4](./墨刀AI用户反馈与满意度体系_V1.md) 的满意度埋点**合并一轮评审**，避免字段冲突。
- 研发侧按"最小可用集"上线，不求一次到位。
- **产出**：埋点 PRD、灰度验证、看板数据接入。

### S3：分群切片（1 周）

- 三维切片：**新/老用户 × 意图场景 × 渠道**。
- 定位 **Top 3 最大掉点**（量级 × 转化差 × 营收弹性 综合打分）。
- **产出**：Top 3 掉点的"诊断卡片"（每张含：现象、量级、假设原因、可行实验方向）。

### S4：体验归因（1–2 周）

- 针对 Top 3 掉点做 **用户旅程回放（埋点事件序列）** + **用户访谈（5–8 人，按激活梯度 L1/L2/L3 抽样）**。
- 把数字翻译成可落地的体验断点。
- **产出**：每个掉点的"断点清单"，每条断点对应至少 1 个改动假设。

### S5：实验循环（每 2 周一轮）

- 每个假设形成一个 **AB 实验卡片**：变更内容、目标指标、护栏指标、最小样本量、运行周期。
- 每轮结束后写实验复盘，胜出的直接扩量，失败的归档经验。
- **产出**：实验看板 + 季度实验汇总。

### S6：看板日常化（长期）

- §7 的 6 张看板 + [反馈体系 §8](./墨刀AI用户反馈与满意度体系_V1.md) 的 3 张看板共 **9 张看板**固化，指定 owner，每周更新解读。
- 关键指标设置阈值告警（如 L3 激活率连续 3 天下跌 >10% 触发自动通知；CSAT 周环比下降 >5pp 触发通知）。
