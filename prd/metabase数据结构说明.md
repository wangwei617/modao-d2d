# Metabase：墨刀 AI 工作台数据结构说明

> **目的**：把 Metabase 导出的「AI 工作台任务」数据，与墨刀 AI 4.0 首页交互（一级 Tab / 二级能力标签 / 对话与生成过程）对齐，便于埋点、分析与看板设计。  
> **样本**：单条对话导出见 `~/Downloads/query_result_2026-04-16T03_39_22.19864Z.csv`（「电视首页生产」案例）；全量导出见仓库内 `query_result_2026-04-16T02_07_33.509669Z.csv`（字段集合可能因查询版本略有差异）。

---

## 1. 数据在描述什么（核心实体）

**一行记录 ≈ 一次「AI 工作台会话/任务」实例**（从用户在首页发起，到工作区内多轮执行、产出文件的整条链路），而不是「一次 HTTP 请求」或「一次模型调用」。

- **产品侧可见**：左侧历史列表标题、对话区、计划/补充信息、生成文件列表与预览、底部「告诉 AI 如何继续…」等。
- **数据侧**：以 `_id` 为主键，关联 `user_id`、组织/空间、工作区路径、计划 `plan`、结果 `work_results`、状态 `status`、Token `token_usage` 等。

## 2. 与首页/对话 UI 的字段映射（你已校准的部分）


| 字段                                          | UI 含义（产品语言）     | 补充说明                                                                                                 |
| ------------------------------------------- | --------------- | ---------------------------------------------------------------------------------------------------- |
| `name`                                      | **对话标题**        | 由系统根据用户 `query` 与上下文**自动归纳**生成，对应左侧历史列表展示名称（如「电视首页生产」）。                                              |
| `mode`                                      | **首页二级能力**      | 对应用户在首页点击二级卡片后，**挂载在输入框左上角的标签**所代表的「能力类型」。在数据里多为枚举字符串（如 `prototyping`），与界面中文文案需一张映射表。                |
| `plan`                                      | **执行计划**        | AI 结合用户所选 `mode` 与输入内容生成的**结构化计划**（多步、子任务、Agent 分工等），对应对话里「计划/补充要点」类信息。序列化形态常为类 EDN/嵌套文本，适合落仓后解析再分析。 |
| `work_results` / `work_results_summary`     | **生成结果**        | 各步骤产出摘要与明细：如创建了哪些文件、预览入口、Agent 名称等。`summary` 常为按步骤索引的浓缩版，体积仍可能很大。                                    |
| `workspace_files`                           | **工作区文件索引**     | 与结果相关的文件元数据列表（路径、类型、分类、时间、`created_by` 等），用于预览、下载与审计。                                                |
| `attachments` / `short_content_attachments` | **用户上传附件**      | 图片/文档等；部分能力（如图片转原型）会与上传强相关。                                                                          |
| `token_usage`                               | **模型 Token 消耗** | 常见为类 map 文本（如 `:prompt_tokens`、`:completion_tokens`、`:total_tokens`），用于成本与强度分析。                      |
| `created_at` / `updated_at`                 | **任务时间**        | 创建与最后更新时间；做「近 30 天」窗口时**以 `created_at` 为准**更贴近「发起需求」口径。                                              |
| `status`                                    | **任务生命周期状态**    | 如 `finished`、`paused`、`error`，以及导出中出现的 `clarifying`（见下文）。                                            |
| `execution_mode`                            | **执行深度/速度策略**   | 与界面「极速」等选项对应的数据侧枚举（如 `express` / `deep`），表示单次任务链路的执行策略，**不同于**首页「一级 Tab / 二级能力」分类。                   |
| `user_id`                                   | **用户**          | 墨刀用户维度主键，用于去重、频次、留存。                                                                                 |
| `org_cid` / `space_cid`                     | **组织 / 空间**     | 团队与企业空间归属；个人空间也可能有对应字段（以实际库表为准）。                                                                     |
| `workspace_path`                            | **服务端工作区根路径**   | 运维与排错价值高于产品分析。                                                                                       |
| `active_transaction_id`                     | **事务 ID**       | 并发、续跑、一致性相关。                                                                                         |
| `resume_step_index`                         | **续跑步骤指针**      | 断点续跑或恢复时落在计划/步骤的哪一步。                                                                                 |
| `initial_generation_completed`              | **首轮生成是否完成**    | 布尔或空；可与 `status` 联立判断「是否已出首版结果」。                                                                     |
| `reason`                                    | **暂停/中断原因文案**   | 在 `paused` 等状态下有意义（如用户主动暂停）。                                                                         |
| `vector_doc_ids` / `retrieved_context`      | **检索增强**        | 绑定向量库文档、检索上下文落盘；可为空。                                                                                 |
| `legacy_project_cid`                        | **旧项目 CID**     | 兼容历史项目体系。                                                                                            |
| `celery_task_id`                            | **异步任务 ID**     | 与后台 Celery 队列追踪相关。                                                                                   |
| `_id`                                       | **记录主键**        | 常见为 MongoDB ObjectId 形态。                                                                             |


### 2.1 关于 `mode = general` 的重要说明

在导出数据中，`general` 往往占大多数。结合首页交互（图 1：未挂载二级标签），更合理的解释是：

- `**general` 表示「未选择或未写入具体二级能力枚举」的默认/通用会话**（用户直接输入、或旧链路未打标）；  
- **不等价于**「用户在 UI 上选择了名为 General 的卡片」。

因此：**用 `mode` 分析「首页二级功能使用率」时，必须把 `general` 单独成类**（未挂载标签 / 默认），避免高估某一垂类能力。

### 2.2 关于 `status = clarifying`

该状态在数据中有可观占比，语义上更接近**「多轮澄清 / 方案对齐阶段」**（例如生成前补充场景、模块、风格等），与图 3 中「补充内容 + 快捷补充按钮」阶段相吻合。是否算「成功完成」取决于看板定义：

- 若 KPI 只认 `finished`，会**低估**仍在澄清链路上的活跃需求；  
- 建议单独监控 `clarifying` 的**转化率**（进入 `finished` 的比例、停留时长、用户是否流失）。

## 3. `mode` 枚举与首页「一级 / 二级」能力的大致对应

首页二级能力见 `墨刀AI4.0改版交互说明.md` 中映射表；数据里 `mode` 为后端枚举，典型对应关系如下（**非穷举，以线上配置为准**）：


| 数据 `mode`（示例）          | 更贴近的二级能力 / 场景              |
| ---------------------- | -------------------------- |
| `prototyping`          | AI 生成原型、图片/HTML 转原型等原型类工作流 |
| `documentation`        | 需求文档 PRD、说明文档类             |
| `competitive_analysis` | 竞品分析                       |
| `test_cases`           | 测试用例生成                     |
| `review`               | 产品方案评审等                    |
| `ui_review`            | UI 规范评审                    |
| `user_research`        | 用户调研                       |
| `product_planning`     | 产品规划                       |
| `tech_design`          | 技术方案生成                     |
| `tech_review`          | 技术方案评估                     |
| `uxspecs`              | 交互/UX 规格类输出                |
| `seo_article`          | SEO 文章                     |
| `image_generation`     | 生图类能力                      |
| `research`             | 泛研究/调研                     |

## 4. 不同 Metabase 导出版本：字段差异说明

你提供的「电视首页生产」单条导出（`~/Downloads/query_result_2026-04-16T03_39_22.19864Z.csv`）比仓库大文件多出若干列，例如：


| 额外字段（示例）                     | 可能用途                                 |
| ---------------------------- | ------------------------------------ |
| `capability`                 | 更细粒度能力码（若线上有「能力 ID」体系，优先用它对齐首页卡片）。   |
| `org_type`                   | 组织类型（如 personal / team）。             |
| `workspace_dirs`             | 工作区目录级索引。                            |
| `suggestions`                | 对话底部**快捷续写/补充**建议（如「补充我的页面」），与转化强相关。 |
| `selected_hot_item_key_list` | 与首页「热门/推荐」运营位选中项相关时的键列表。             |
| `modification_context`       | 二次修改上下文（用于迭代编辑类能力）。                  |
| `flpak_key`                  | 与工作区打包/同步相关的内部键（以工程定义为准）。            |


**结论**：Metabase 问法不同，列集合会不同。做公司级数据字典时，建议以**数仓表 DDL**为准，并把「Metabase 问题 → 底层表 → 字段」写成一页索引，避免各导出口径混用。

## 5. 分析口径建议（给数据与产品的共同约定）

1. **时间窗口**：近 N 天默认用 `created_at` 过滤；若分析「最后活跃」用 `updated_at`。
2. **用户去重**：`user_id` distinct count。
3. **能力使用率**：`mode` 分组时**单独展示 `general`**，其余为「显式选择能力」。
4. **成功与失败**：
  - `finished`：主成功口径；  
  - `error`：失败；  
  - `paused`：结合 `reason` 区分用户主动 vs 系统/ worker 问题；  
  - `clarifying`：澄清中，建议做漏斗而非简单成功/失败。
5. **成本与强度**：解析 `token_usage.total_tokens`，注意长尾任务会极端拉高均值，建议同时给出 **P50 / P90**。
6. **文本字段**：`plan`、`work_results`*、`workspace_files` 体积大，**不适合**直接在 BI 里做全文本聚合；应先解析结构化字段或落特征（文件数、步骤数、是否含错等）。
