# 墨刀 AI 用户反馈与满意度体系 V1

> **文档类型**：用户反馈采集 × 满意度评价方法论与执行路径
> **版本**：v1.0（2026-04-17）
> **目标**：建立一套**定性 × 定量双轮驱动**的反馈分析体系，回答数据漏斗无法回答的"为什么"，尤其服务营收主战场——激活 L2→L3 与付费决策。

## 0. TL;DR

- **定位**：反馈体系不是客服流程的升级，而是**增长分析的并行支柱**，与数据漏斗双轮驱动。
- **核心模型**：双数据源（网易七鱼工单 + 产品内嵌评价）→ 三层评价（会话 CSAT / 环节评分 / 整体 NPS）→ 四类分类（用户原因 / 内部根因 / 环节归属 / PRD 映射）→ 双闭环（个人 + 运营）。
- **采集原则**：只在 **7 个决定性瞬间**采集，不做全域骚扰。
- **打通要求**：所有反馈必须关联 `user_id`，打通神策事件 / Metabase 业务表 / 客服工单，实现「**反馈 × 激活梯度 × 付费状态**」的交叉分析。
- **Owner**：客服主理（运营侧），PM 与交互团队参与分类体系与闭环动作。
- **MVP 范围**：只做采集时机 1 / 4 / 6（任务完成后 / 导入墨刀后 / 支付前后）+ 工单打通 + 差评热点周报，3 个月后评估扩量。

## 1. 现状盘点与体系定位

### 1.1 现状

| 维度 | 现状 | 问题 |
|---|---|---|
| **反馈来源** | 网易七鱼客服工单 | 用户主动联系才有，沉默大多数数据盲区 |
| **满意度评价** | 无产品内嵌评价 | 无法量化满意度、无法分群、无法对比版本 |
| **反馈与数据** | 工单独立系统，不与神策/Metabase 打通 | "反馈热点 vs 漏斗掉点"无法交叉 |
| **分类方式** | 客服口径分类，工程视角缺失 | 难以直接转化为产品迭代输入 |
| **闭环** | 工单处理闭环到客服，产品侧弱 | 用户不知道自己反馈是否被采纳 |

### 1.2 为什么定位为"增长分析的并行支柱"

数据告诉你"哪里掉了"，反馈告诉你"为什么掉"。在墨刀 AI 当前阶段，下列场景**定量数据近乎失效**，必须靠定性反馈补齐：

| 场景 | 定量能看到 | 定性必须回答 |
|---|---|---|
| 激活 L2→L3 掉点 | 多少用户没触发 L3 | 看到产出后为什么不导入墨刀 |
| 付费关闭 | 支付页放弃率 | 看到价格为什么关掉 |
| 错配治理（R1） | 队内错配率 55.3% | 用户是点错还是表达错 |
| 流失（Churn） | D30 不回访比例 | 最后一次使用的印象是什么 |
| Top4 场景深化（R3） | 各场景 `finished` 率 | 产物为什么"看着对却用不上" |

## 2. 反馈数据源与采集模型

### 2.1 双数据源架构

```
           ┌──────────────────────────┐
           │  数据源 A：网易七鱼工单   │
           │  （存量 + 高价值深度反馈） │
           └────────────┬─────────────┘
                        │ API 同步
                        ▼
           ┌──────────────────────────┐     ┌──────────────────────┐
           │    反馈数据湖（Feedback  │ ←── │  数据源 B：产品内嵌   │
           │       Data Lake）         │     │  （CSAT/环节评分/NPS）│
           └────────────┬─────────────┘     └──────────┬───────────┘
                        │                              │ 神策埋点
                        │                              │
                        │         ┌────────────────────┘
                        ▼         ▼
                 ┌──────────────────────┐
                 │ 通过 user_id 打通    │
                 │ 神策事件 / Metabase  │
                 │ 业务表 / 订单表      │
                 └──────────────────────┘
                        │
                        ▼
                 定性 × 定量 交叉分析
```

### 2.2 两条数据流的职责划分

| 数据源 | 承担 | 样本特点 | 分析用途 |
|---|---|---|---|
| **网易七鱼工单** | 高价值深度反馈、投诉、功能建议、bug 报告 | 少而重，文字长 | 深度根因挖掘、PRD 需求输入 |
| **产品内嵌评价** | 高频结构化反馈（👍👎 / 星级 / 标签） | 多而轻，标签化 | 漏斗级满意度监控、趋势预警、A/B 实验对照 |

两者**不是替代关系，是互补**：工单是"望远镜看到远处但密度低"，内嵌评价是"扫描仪全面但浅"。

## 3. 三层满意度评价体系

### 3.1 整体设计

| 层级 | 名称 | 频率 | 形式 | 样本量 | 信息深度 |
|---|---|---|---|---|---|
| **L1** | 会话级 CSAT | 每次任务完成 | 👍 👎 +（若 👎）快捷原因标签 | 高 | 浅 |
| **L2** | 环节级评分 | 关键动作后 | 1~5 星 + 分类标签 + 可选文字 | 中 | 中 |
| **L3** | 整体 NPS | 每 30/60 天 / 或达使用阈值 | NPS（0-10）+ 原因 + 改进建议 | 低 | 深 |

### 3.2 三层关系

- **L1 是"脉搏"**：每天可监控趋势、版本对比、场景对比。
- **L2 是"靶点"**：在关键营收/体验节点精准打样，驱动局部优化。
- **L3 是"体检"**：衡量长期关系质量，用于战略判断与对外披露。

**关键原则**：L1 和 L2 触发频率有限制，不重叠、不骚扰（参见 §4 采集时机）。

### 3.3 指标定义

| 指标 | 定义（分子 / 分母） | 阈值建议 |
|---|---|---|
| **CSAT（L1）** | 👍 数 / （👍 + 👎）| > 85% 健康，< 75% 预警 |
| **👎 率** | 👎 数 / 有评价任务数 | < 10% 健康，> 15% 预警 |
| **环节评分（L2）** | 该环节 1~5 星均分 | > 4.2 健康，< 3.8 预警 |
| **NPS（L3）** | 推荐者（9-10）% − 批评者（0-6）% | > 30 优秀，0~30 合格，< 0 危险 |

## 4. 采集时机：7 个决定性瞬间

不要在所有地方都问。**只在下列 7 个时机采集**，触发频率统一控制在「**同一用户 24 小时内最多 2 次、同类型评价 7 天内最多 1 次**」。

| # | 时机 | 对应环节 | 层级 | 核心问题 | 优先级 |
|---|---|---|---|---|---|
| 1 | 首次任务 `finished` 后（沙盒加载完成） | 激活 L1→L2 | L1 CSAT | 第一印象是否达标 | **P0（MVP）** |
| 2 | 触发错配提示并用户做出选择后 | R1 错配治理 | L1 CSAT（建议有用/没用） | 错配识别是否准 | P1 |
| 3 | `clarifying` 阶段完成后 | R5 澄清 | L1 CSAT | 澄清是否消耗体验 | P2 |
| 4 | **导入墨刀后 3 分钟内**（或首次编辑动作后） | 激活 L3 | L2 评分 | 还原度 + 编辑顺畅度 | **P0（MVP）★** |
| 5 | 分享链接被外部访问 ≥1 次后回访用户 | 激活 L3 + 裂变预埋 | L2 评分 | 分享场景是否达预期 | P1 |
| 6 | **支付成功后 / 支付页关闭离开前（意图明显）** | 付费决策 | L2 评分 + 原因 | 付费动因 & 放弃原因 | **P0（MVP）★** |
| 7 | 30 天内会话 ≥ 3 次的活跃用户 | 留存 + 整体健康度 | L3 NPS | 长期关系质量 | P1 |

**★ 标注**：时机 4 和 6 是营收决策最关键的两个采集点，MVP 必须包含。

### 4.1 各时机话术与交互建议

#### 时机 1：首次任务完成后（L1 CSAT）

**触发条件**：`status=finished` 且是该用户 7 天内第一次首任务完成。
**UI**：沙盒右上角轻量浮层，3 秒自动淡出或点击外部关闭，不阻断操作。
**文案**：
> 这次生成符合你的预期吗？  
> 👍 满意｜👎 不满意

**点击 👎 后弹出快捷原因**（单选 + 可选"其他"自由文本）：
- 理解错了我的意思
- 产出太粗糙/不能用
- 样式不符合预期
- 多文件结构乱
- 想要的功能不支持
- 其他（100 字内）

---

#### 时机 4：导入墨刀后 3 分钟内（L2 评分）★

**触发条件**：用户完成"导入墨刀 → 编辑器内保存"后 3 分钟，或首次编辑动作后。
**UI**：墨刀编辑器内弹出非阻断性问卷卡片。
**文案**：
> 刚刚导入的 AI 原型，和你期待的一致吗？  
> ⭐⭐⭐⭐⭐（1~5 星）

**评分 ≤ 3 星时展开分类**（多选）：
- 结构和 AI 生成时不一致
- 组件不能编辑/不规范
- 样式丢失或错乱
- 导入太慢
- 不知道怎么继续编辑
- 其他（200 字内）

---

#### 时机 6：支付场景（L2 评分 + 原因）★

**6A 支付成功后**：
> 感谢支持！一个快速问题——你决定付费最主要的原因是？（单选）
> - AI 生成次数不够用
> - 要把作品导入墨刀继续完善
> - 要分享/发布给他人看
> - 需要更高级的模型/深度模式
> - 限时特惠价格合适
> - 其他

**6B 支付页关闭前（意图识别）**：
> 离开前告诉我们一下——没有选择付费的主要原因是？（单选，可跳过）
> - 价格偏高
> - 没找到适合我的套餐
> - 暂时还用不上
> - 担心积分用不完
> - 想先对比其他产品
> - 其他

**关键埋点**：此时必须记录 `paywall_trigger_source`、`plan_viewed`、`dwell_time`，与关闭原因一起落表。

---

#### 时机 7：活跃用户 NPS（L3）

**触发条件**：30 天内会话 ≥ 3 次、且未做过 NPS（60 天冷却期）。
**UI**：首页横条问卷，可关闭。
**文案**：
> 根据你这段时间的使用体验，有多大可能把墨刀 AI 推荐给同事或朋友？  
> （0 = 绝不推荐，10 = 极力推荐）  
> 请补充一句最想告诉我们的话：____（可选）

**分档处理**：
- 9-10（推荐者）：触发邀请码/分享引导，反向做裂变钩子
- 7-8（中立者）：不做额外动作
- 0-6（批评者）：客服主动联系 + 产品跟进

## 5. 反馈分类体系：双轨分类

### 5.1 第一轨：用户可见的"原因标签"

控制在 5~7 个选项，保证用户 3 秒内能选完。分类按环节设计，参见 §4 各时机话术。

### 5.2 第二轨：内部的"根因标签体系"（运营分析用）

用户选完原因后，**运营/产品侧做二次标注**，映射到 R1~R7 与数据漏斗环节：

| 用户原因 | 内部根因标签 | 归属环节 | 对应 PRD |
|---|---|---|---|
| 理解错了我的意思 | 意图识别失败 | 激活 L1→L2 | R1 / R2 |
| 产出太粗糙 | 模型能力 / 场景深度不足 | 激活 L2 | R3 |
| 样式不符合预期 | 模型风格偏好 | 激活 L2 | R3 |
| 多文件结构乱 | 多文件工程化弱 | 激活 L2→L3 | R3 |
| 结构和 AI 生成时不一致 | 导入映射失真 | 激活 L3（导入） | R3 |
| 组件不能编辑/不规范 | 组件映射策略 | 激活 L3（导入） | R3 |
| 导入太慢 | 导入性能 | 激活 L3（导入） | R3 / 性能护栏 |
| 不知道怎么继续编辑 | 引导缺失 | 留存 | R4 |
| 功能没找到 | 入口问题 | 激活 L1 | R4 / R6 |
| 报错/失败 | 技术异常 | 全环节 | R2 |
| 澄清太多次 | 澄清体验 | 激活 L1→L2 | R5 |
| 价格偏高 | 定价策略 | 付费 | 付费（新增） |
| 套餐不合适 | 套餐设计 | 付费 | 付费（新增） |
| 积分用不完 | 额度设计 | 付费 / 留存 | 付费（新增） |

### 5.3 分类体系维护规则

- **版本化**：分类体系本身版本化管理，V1/V2 迭代在本文 §5.2 追加 patch 记录。
- **新增门槛**：某个自由文本聚类频次 ≥ 20 次/月、占比 ≥ 3%，考虑提升为固定分类。
- **合并/下线**：连续 2 个月频次 < 5 次 或占比 < 0.5% 的固定分类，考虑合并或下线。
- **Owner**：客服主运营分类库，PM 每月复核。

## 6. 数据打通方案

### 6.1 三方数据打通架构

```
┌─────────────────┐   ┌──────────────────┐   ┌─────────────────┐
│  网易七鱼工单    │   │   产品内嵌评价    │   │    神策事件       │
│  （客服/工单库） │   │  （CSAT/星/NPS）  │   │  （漏斗/路径）    │
└────────┬────────┘   └─────────┬────────┘   └────────┬────────┘
         │ API 同步             │ 埋点上报            │ 事件流
         │ (每日/实时)          │                     │
         ▼                      ▼                     ▼
┌──────────────────────────────────────────────────────────────┐
│        反馈数据湖（建议落到数仓 / PG）                        │
│        主键：user_id | org_cid | session_id | ticket_id       │
└────────┬──────────────────────────────┬──────────────────────┘
         │                              │
         │                              │
         ▼                              ▼
┌──────────────────┐           ┌──────────────────┐
│   Metabase       │           │  神策（事件层）   │
│  - 反馈列表        │           │  - 漏斗节点归因   │
│  - 工单 × 订单交叉 │           │  - 反馈事件 × 转化│
│  - 分类趋势        │           │                  │
└──────────────────┘           └──────────────────┘
```

### 6.2 打通的核心字段

所有反馈事件/工单必须包含：

| 字段 | 来源 | 用途 |
|---|---|---|
| `user_id` | 各源 | 主键打通 |
| `org_cid` | 工单/订单 | 空间维度（B 端客户识别） |
| `session_id` / `task_id` | 产品内嵌（神策） | 关联具体任务 |
| `ticket_id` | 网易七鱼 | 工单唯一标识 |
| `event_name` | 内嵌（神策） | 如 `feedback_csat_submitted` |
| `feedback_layer` | 各源 | L1/L2/L3 枚举 |
| `trigger_moment` | 内嵌（神策） | 对应 §4 的 7 个时机编号 |
| `user_reason_tag` | 各源 | 用户原始选择 |
| `internal_root_cause` | 运营标注 | R1~R7 映射 |
| `text_feedback` | 各源 | 自由文本原文（脱敏前保留） |
| `created_at` | 各源 | — |

### 6.3 网易七鱼工单接入方案

**目标**：把工单数据同步到反馈数据湖，与内嵌评价、神策事件关联。

| 步骤 | 内容 | Owner |
|---|---|---|
| 1 | 获取网易七鱼 API Key（需客服账号管理员权限） | 客服 + IT |
| 2 | 工单字段映射（工单 → 反馈数据湖 schema） | 数据 |
| 3 | 每日增量同步脚本（定时任务 T+1 即可起步） | 数据 |
| 4 | 用户身份匹配：工单中的用户信息（邮箱/手机/user_id）→ 墨刀 user_id | 数据 + 后端 |
| 5 | 客服二次标注流程：工单关闭时，客服在工单中打「内部根因标签」下拉框 | 客服 |
| 6 | 同步到 Metabase，建立"工单看板" | 数据 |

**关键决策**：工单的「根因打标」是否由客服完成、还是 PM 复标？
- 建议：**客服在工单关闭时做初标**（必填项，不关闭无法提交），**PM 每周复标/修正**（保证口径稳定）。

### 6.4 产品内嵌评价走神策埋点

**事件命名规范**（建议）：

| 事件名 | 时机 | 关键属性 |
|---|---|---|
| `feedback_csat_shown` | 时机 1/2/3 浮层展示 | `trigger_moment`, `task_id`, `mode`, `scene` |
| `feedback_csat_submitted` | 用户点击 👍/👎 | 上同 + `sentiment`, `reason_tag`, `text` |
| `feedback_stage_rating_shown` | 时机 4/5/6 问卷展示 | `trigger_moment`, `related_action_id` |
| `feedback_stage_rating_submitted` | 用户提交星级 | 上同 + `rating`, `reason_tag`, `text` |
| `feedback_paywall_exit_shown` | 时机 6B 关闭前意图识别 | `paywall_trigger_source`, `plan_viewed` |
| `feedback_paywall_exit_submitted` | 用户选择原因 | 上同 + `exit_reason_tag` |
| `feedback_nps_shown` | 时机 7 NPS 横条展示 | `session_count`, `days_since_first` |
| `feedback_nps_submitted` | 用户提交 NPS | 上同 + `score`, `comment` |

## 7. 运营闭环

### 7.1 个人闭环（对单个用户）

**目标**：让用户知道"反馈有人看、有人处理"，把批评者变推荐者。

| 反馈层级 | 动作 | Owner | SLA |
|---|---|---|---|
| 工单（严重投诉/bug） | 客服当日响应 + 产品 3 日内给出处理计划 | 客服 → PM | T+1 响应 / T+3 计划 |
| 工单（功能建议） | 客服回复"已收到，会评估" + 入评估池 | 客服 | T+1 |
| L1 👎 高频用户 | 客服主动外呼/站内信 | 客服 | 每周抽样 20% |
| L2 ≤ 3 星且附文字 | PM 阅读 + 若命中已规划需求，告知用户预期上线 | PM | 每周一次 |
| L3 NPS 批评者（0-6） | 客服主动联系 + 了解原因 | 客服 | 每月一次 |
| **重大改进上线** | 站内信/邮件触达曾反馈此问题的用户："你提到的 XX 已改进" | 产品运营 | 随版本 |

### 7.2 运营闭环（对产品决策）

**目标**：让反馈数据持续灌入产品决策，不做信息孤岛。

| 周期 | 产出 | 受众 | Owner |
|---|---|---|---|
| **每周** | 《用户反馈周报》：Top 10 反馈热点、按环节归类、相关漏斗数据 | PM / 设计 / 研发 / 商业化 | 客服 + PM 联合 |
| **每双周** | 《反馈 × 实验》对照：本周期实验上线后，相关环节的反馈评分变化 | 增长团队 | PM |
| **每月** | 《满意度健康度》：CSAT / NPS / 环节评分趋势 + 异常归因 | 全体 | PM |
| **每季度** | 《反馈驱动 PRD 输入》：本季反馈沉淀下来、进入 PRD 的需求清单 | 立项委员会 | PM |

## 12. 非目标与注意事项

- **不做"全场景评分弹窗"**：过度骚扰是满意度体系最大杀手，仅限 §4 的 7 个时机。
- **不追"高 NPS 本身"**：NPS 是健康度体检，不是 KPI，避免刷分。
- **不把客服工单当全部**：工单只覆盖主动联系用户，沉默大多数要靠内嵌评价覆盖。
- **不在 MVP 阶段做 AI 自动分类**：先靠人工 + 规则，数据量稳定后再考虑。

## 附：关键术语

- **CSAT (Customer Satisfaction Score)**：客户满意度评分，本文 L1 层级使用 👍👎 二元形式。
- **NPS (Net Promoter Score)**：净推荐值，推荐者占比 − 批评者占比。
- **反馈数据湖**：本文指统一存储工单 / 产品内嵌 / 神策事件的数据层，物理形态可以是数仓一个 schema、PG 一组表，或两者组合。
- **反馈 × 漏斗交叉**：本文核心方法，把满意度评分与转化漏斗节点在看板上同屏对照，用相关性判断"体验恶化是否引发转化下降"。
